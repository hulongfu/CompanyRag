# 文档解析与切分

## 概述

文档模块负责文档上传、文本提取、智能切分和向量化存储。核心流程：

```
文件上传 → Tika 解析 → 智能切分 → 向量化 → PGVector 存储
```

## 文档处理流程

`DocumentParseServiceImpl.uploadAndParse()` 完整流程：

1. **保存文档元数据** — 写入 `rag_document` 表，状态为"待处理"(0)
2. **文本提取** — Apache Tika 自动检测文件类型并提取纯文本
3. **智能切分** — 默认使用 `SEMANTIC_CHUNK` 策略，chunkSize=512，overlap=64
4. **保存切分块** — 写入 `doc_chunk` 表
5. **向量化存储** — 调用 Embedding 模型生成向量，存储到 PGVector
6. **更新文档状态** — 状态为"已向量化"(2)

失败时状态设为 -1，并记录错误信息。

## 文本提取

使用 Apache Tika 3.1.0，支持 PDF/DOCX/TXT/MD/HTML 等格式。

TXT 文件特殊处理：检测提取率（文本长度/文件大小），若低于 80% 则尝试 UTF-8 直接读取。

## 切分策略

### 策略对比

| 策略 | 枚举值 | 原理 | 适用场景 | Token 利用率 |
|------|--------|------|---------|------------|
| 语义切分 | SEMANTIC_CHUNK | 按 Markdown 标题/段落边界递归切分 | 结构化文档 | ⭐⭐⭐⭐⭐ |
| 滑动窗口 | SLIDING_WINDOW | 固定大小 + 重叠 + 句边界感知 | 通用文本 | ⭐⭐⭐⭐ |
| 固定大小 | FIXED_SIZE | 按字符数切分 | 无结构文本 | ⭐⭐⭐ |

### 语义切分器 (SemanticChunkSplitter)

两阶段切分：

**第一阶段**：按语义边界粗切
- 按 Markdown 标题（`#` ~ `######`）分割
- 保留代码块完整性

**第二阶段**：对过长段落实行递归细切
- 按段落（双换行）切分
- 保留重叠区域避免信息断裂
- 强制字符数上限（400 字符 < 512 tokens）
- 在句子边界处断开（`.。!！?？`）

### 固定大小切分器 (FixedSizeSplitter)

- 按固定字符数切分，无重叠
- 安全字符数上限 400（防止 token 超限）

### 滑动窗口切分器 (SlidingWindowSplitter)

- 固定窗口大小 + 重叠区域
- 句边界感知，尽量不在句子中间断开

## 向量化存储

`vectorizeAndStore()` 方法：

1. 为每个 chunk 构建元数据（documentId、tenantId、documentName、chunkIndex）
2. 使用 `UUID.randomUUID()` 生成向量存储 ID（Spring AI PgVectorStore 要求 UUID 格式）
3. 批量调用 `vectorStore.add()` 向量化并存储

## 数据模型

### Document

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Long | 主键 |
| tenantId | Long | 租户 ID |
| fileName | String | 文件名 |
| fileType | String | 文件类型 |
| fileSize | Long | 文件大小 |
| chunkCount | Integer | 切分块数量 |
| status | Integer | 状态(-1失败/0待处理/1已切分/2已向量化) |
| errorMsg | String | 错误信息 |

### DocumentChunk

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Long | 主键 |
| documentId | Long | 所属文档 ID |
| tenantId | Long | 租户 ID |
| chunkIndex | Integer | 块序号 |
| content | String | 文本内容 |
| tokenCount | Integer | Token 估算数量 |
| splitStrategy | String | 切分策略名称 |
