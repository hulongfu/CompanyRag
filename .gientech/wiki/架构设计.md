# 架构设计

## 分层架构

```
┌──────────────────────────────────────┐
│         Bootstrap (启动层)            │  全局配置、Bean 注册、应用入口
├──────────────────────────────────────┤
│           Web (表现层)                │  REST Controller、页面路由
├──────────────────────────────────────┤
│       Agent (智能编排层)              │  MCP 工具编排、意图识别
├──────────────────────────────────────┤
│         RAG (核心业务层)              │  混合检索、Rerank、缓存、Prompt
├──────────────────────────────────────┤
│  Tenant (租户层) │ Document (文档层)  │  租户隔离 │ 文档解析/切分/向量化
├──────────────────────────────────────┤
│         Common (基础层)               │  常量、异常、统一响应、工具类
└──────────────────────────────────────┘
```

## 模块依赖方向

```
common ← tenant ← document ← rag ← agent
                             ↑
                    web ←──────┘
                             ↑
                    bootstrap (依赖所有模块)
```

**依赖规则**：
1. `common` 无项目内部依赖
2. `tenant` 依赖 `common`
3. `document` 依赖 `common`
4. `rag` 依赖 `document` + `tenant` + `common`
5. `agent` 依赖 `rag` + `common`
6. `web` 依赖 `rag` + `agent` + `document` + `common`
7. `bootstrap` 依赖所有模块

**禁止循环依赖和跨层越级调用**。

## 数据流

### RAG 检索数据流

```
用户请求 → Controller (参数校验) → Service (业务编排) →
  ├─ 缓存命中 → 直接返回
  └─ 缓存未命中 →
       ├─ 向量检索 (PGVector HNSW)
       ├─ 关键词检索 (pg_trgm)
       ├─ 融合排序
       ├─ Cross-Encoder Rerank
       ├─ LLM 生成 (通义千问)
       └─ 写入缓存 + 记录指标
```

### 文档处理数据流

```
文件上传 → Tika 解析 → 智能切分(SemanticChunk) → 
  ├─ 保存文档元数据 → rag_document
  ├─ 保存切分块 → doc_chunk
  └─ Embedding 向量化 → PGVector (vector_store)
```

### Agent 编排数据流

```
用户提问 → LLM 意图分析 → 是否需要工具？
  ├─ 不需要 → LLM 直接回答
  └─ 需要 → 选择工具 → 执行工具 → 结果反馈 LLM → 生成最终回答
```

## 事务边界

- 每个 Service 方法为一个事务单元
- 跨模块调用不共享事务
- 缓存操作不在事务内执行
- LLM 调用不在事务内执行

## 外部依赖

| 外部系统 | 访问方式 | 配置位置 | 熔断保护 |
|---------|---------|---------|---------|
| PostgreSQL + PGVector | Spring Data + MyBatis-Plus | application.yml datasource | 连接池 Hikari |
| Redis | Redisson Client | application.yml redis | Redisson 自带重试 |
| 通义千问 DashScope | Spring AI OpenAI 兼容模式 | application.yml ai.openai | Resilience4j CircuitBreaker |
| Prometheus | Micrometer + Actuator | application.yml management | 无（监控系统） |

## 设计原则

1. **模块边界清晰，依赖单向** — 各模块职责单一，通过 Maven 模块依赖关系体现架构分层
2. **多租户安全是首要约束** — Schema 物理隔离 + 行级安全策略双重保障
3. **可观测性是默认能力** — 核心业务流程必须埋点：请求数、延迟、召回率、Token 消耗
4. **防御性编程** — 外部依赖必须有熔断保护，全局异常处理器兜底
5. **AI 集成抽象** — 通过 Spring AI 统一抽象，不直接依赖特定厂商 SDK
